Соглашения о программном коде

Документ описывает соглашения о программном коде, принятые командой разработчиков Testing Suite и предназначен для программистов и проектировщиков системы, занятых над этим проектом. Данных соглашений следует придерживаться при составлении документов, написании кода и обмена информацией.

Соглашения о документах

В данном разделе мы опишем основные правила составления документов. Этот раздел будет полезен прежде всего составителям спецификации и документа, описывающего проектирование системы и алгоритмы использующиеся в ней. Так как эти документы составляются несколькими людьми, необходимо стандартизовать названия файлов, ввести правила написания заголовков, обозначения истории изменения и номеров версий.

Форматы документов

Стандартным форматом документов является формат HTML. Этот формат наиболее распространен в мире. Редактирование такого документа дело не простое, но зато текстовый формат хранения намного лучше бинарного (Word, OpenOffice) так как позволяет отслеживать изменения версий с помощью Subversion. Для редактирования можно использовать как Microsoft Word, так и любой другой текстовый редактор. В качестве шаблона может служить этот документ.

Следует также учесть, что весь текст пишется в кодировке UTF-8.

Версии документов

Версия документа в имени файла не указывается. Документы хранятся в репозитории, что позволяет легко отслеживать изменения и иметь доступ к последней версии документа.

Соглашения о программном коде

В этом разделе говорится о соглашениях, принятых командой разработчиков Steel Engine.

Кодировка

Исходный код пишется в кодировке UTF-8. В начале файла не должны присутствовать символы c кодами 239, 187, 191. Символы  кодом больше, чем 127 могут присутствовать только в комментариях.

переводы строк

В конце последней строки должен быть символ перевода строки.

Названия файлов

Все файлы и директории должны состоять только из маленьких английских букв и цифр. Если в названии несколько слов, то следует отделять их знаком подчёркивания (_). Например, steel_engine.h

В ссылках на модуль указывается относительный путь к файлам без расширения. К модулю относятся все файлы с таким именем. Например, к модулю graph относятся файлы graph.h и graph.cpp. Это позволяет изменять внутренний формат хранения файлов (в основном за формат хранения отвечает расширение файла) без изменения ссылок на этот модуль (файл).

Заголовки файлов

В каждом модуле хранится заголовок с номером версии модуля и информацией об авторах (и, возможно, лицензионном соглашении). Заголовок должен находиться в самом начале файла, описывающего интерфейс модуля, иметь вид комментария, и следующий формат:

 

/*id*********************************************************

    File: <путь к файлу относительно корня исходников>

    Unit: <название модуля>

    Part of: <название проекта>

   (C) DiVision, <years>

    Authors:

        * <автор1>: <его заслуга>

        ...

        * <авторN>: <его заслуга>

    License:

        Steel Engine License

    Description:

        <Непосредственно описание модуля>

    [TODO]

        ...

    [Parts:]

        <Список подмодулей>

************************************************************/

 

Где <автор1> - <nick> [Full Name, email, homepage]

<years> - время разработки именно этого файла.

<название проекта> = Steel engine

<название модуля> пишется маленькими буквами. Если модуль является подмодулем, то названия модулей разделяются / (по аналогии с директориями).

 

Не обязательной является секция Part of, если модуль не является непосредственно частью проекта и может быть использован в других целях.

Также не обязательной является секция TODO.

Changelog не нужно указывать в файле, так как все изменения хранятся в репозитории.

Требования к программному коду (C++)

В этом подразделе перечислены требования к коду для программ на C++, PHP и JAVA, а также их стилевое оформление. Мы рассмотрели несколько стилей, существующих в мире сообщества C++ и JAVA и создали свой стилевой стандарт. Это позволит читать исходные тексты других членов команды не отвлекаясь на разбор стиля и отделение блоков. Некоторые вещи позаимствованы из стиля ANSI C, некоторые — из стиля Kernighan & Ritchie и JAVA, что-то — из примеров книги Brain W. Kernighan и Rob Pike.

При написании кода следует учитывать, что читабельность важнее эффективности. Не следует употреблять сомнительные и неопределенные стандартами конструкций языка. Следует заботиться также о переносимости программы на другие платформы. Наша цель — получить код, который можно перенести на другую платформу практически без изменений.

Если же эффективность в данном месте важнее читаемости, то следует в комментариях оставить читаемую версию кода.

Отступы и разделители

Отступы используются для сдвига содержимого блоков кода. Они делаются символом табуляции, причем он не должен в последствии заменяться пробелами. Это позволяет настроить размер такого отступа и отображать код в соответствии с индивидуальными предпочтениями каждого программиста в команде. Визуально по умолчанию один символ табуляции равняется 4ём пробелам.

Основные части программы (например, реализации двух функций или методов класса) следует отделять друг от друга пустыми строками. Пустой строкой можно отделять различные по смыслу блоки (наборы функций в дефиниции класса, большие циклы внутри процедуры).

Параметры функций отделяются друг от друга запятыми, сразу после которых ставится один пробел. После открывающей скобки и перед закрывающей пробелы не ставятся.

Слева и справа от оператора присваивания следует ставить пробелы.  Можно также отделять пробелами и другие операции, но это следует делать так, чтобы программист мог легко распознать формулу и порядок действий.

В некоторых случаях может потребоваться написать очень большую формулу или длинное условие, которое не помещается на одну строку. Тогда мы можем разбить большое выражение на строки.

Например

(((c1)&&((c2)||(c3))&&(!((c4)&&((c5)||(c6)))))

превращается в

(

  (cccccccccccccc) && ((cccccccccc)|(cccccccccccccc))

  &&

    !(

      (ccccccccccccccccccccc)

      &&

      (

       (cccccccccccccccccccccccccccccccccccccccc)

       ||

       (cccccccccccccccccccccccccccccccccccccccc)

      )

    )

)

Обычно длина строки не должна превышать 80 символов.

Неполнота разделенных строк должна быть очевидной.

totalSum = a + b + c +

           d + e;

Желательно экранировать символ перевода строки, для того, чтобы одним однострочным комментарием можно было закомментировать весь оператор.

function (param1, param2, \

          param3);

Includes

Include-выражения должны быть отсортированы и сгруппированы. Сортировка происходит по их иерархическому положению в системе, при этом низкоуровневые файлы прикрепляются первыми.

 

#include <fstream>

#include <iomanip>

 

#include <qt/qbutton.h>

#include <qt/qtextfield.h>

 

#include "com/company/ui/PropertiesDialog.h"

#include "com/company/ui/MainWindow.h"

 

Include-выражения должны находиться только вначале файла.

Блоки

Знаки обозначающие начало и конец блока (обычно, это — фигурные скобки) могут располагаться двумя способами:

1.      на отдельных строках (стандарт ANSI);

2.      открывающий знак пишется на строке, где начинается инструкция, содержащая данный блок; закрывающий знак - на отдельной строке (стиль Kernighan & Ritchie).

3.      весь блок пишется после конструкции (inline).

Рекомендуется использовать первый способ. Весь код блока пишется с отступом.

Комментарии

Комментарии в программе пишутся на русском или английском.

Короткие комментарии (обычно размером не более 3-х строк) пишутся с использованием знака (token), обозначающего комментарий, продолжающийся до конца строки (обычно это 2 слэша). Такие комментарии описывают небольшой объект, и отделяется от текста  программы пробелом или символом табуляции. Сразу после знака комментария ставится один пробел, за которым следует текст комментария. Если комментарий продолжается на следующей строке, то новый знак комментария ставится под предыдущим (на следующей строке), после чего ставится пробел, за которым комментарий можно продолжить.

Пример:

printf("Hello, world!"); // this is the comment for

                         // function printf()

Длинные комментарии (обычно размером от 3-х строк) пишутся с использованием знаков, открывающих и закрывающих комментарий, которые пишутся на новой строке с тем же отступом, что и последующая инструкция. Между знаками с таким же отступом пишется текст комментария. Такой комментарий обычно используется для описания функций.

Пример:

/*

    This is a very very very very very very very very

    very very very very very very very very very

    very very very very very very very very very

    very long comment for the function printf().

*/

printf("Hello, world!");

 

Следует заметить, что короткие комментарии пишутся перед или после элемента кода, который они поясняют, а длинные – перед поясняемым кодом.

 

Если следует закомментировать большой участок кода для того, чтобы он не компилировался, лучше воспользоваться директивой #if 0, чтобы не получилось конфликта вложенных комментариев.

Идентификаторы

Названия переменных, функций и типов пишутся на английском языке. По возможности, следует также использовать традиционные обозначения, например, переменную i в цикле for. Желательно не использовать сокращения в идентификаторах (кроме общепринятых, вроде init, mem).

computeAverage();   // NOT: compAvg();

HTML // NOT: HypertextMarkupLanguage

Избегайте отрицательных логических выражений. Положительная логика лучше отрицательной.

bool isError; // NOT: isNoError, а еще лучше isAllOk

bool isFound; // NOT: isNotFound

Глобальный идентификатор не должен совпадать с каким-либо идентификатором из используемых библиотек или других модулей. При необходимости, перед идентификатором добавляется имя модуля для уникальности идентификатора.

Если в начале идентификатора должно стоять число, то в начало добавляется подчёркивание.

All names should be written in English.

filename   // NOT: imjaFaila

Иногда можно отступать от стандарта, если отступление повысит читаемость кода или решение по стандарту неэффективно. Цель стандарта – повысить читаемость кода.

Константы

Названия констант и переменных, которым присваивается значение только 1 раз и они действуют на всю программу, пишутся большими буквами. Слова разделяются знаком подчёркивания.

const int MAX_N = 5;

#define MAX_N 5

Сюда относятся и значения перечисляемых типов. В начале каждой константы должен быть [сокращённый] идентификатор типа, которому он принадлежит. У всех констант одного типа этот префикс должен совпадать.

typedef enum

{

  POSITION_NONE,

  POSITION_LOCAL,

  POSITION_GLOBAL

} PositionKind;

Переменные

Переменные должны начинаться с маленькой буквы. Первая буква каждого внутреннего слова должна быть большой. Допускается написание переменных со всеми маленькими буквами. В этом случае слова отделяются подчёркиванием.

line, myAccount, centerOfGravity, theRectangle

point_count

Если переменная является массивом (или содержит множество других объектов, вроде set или map), то имя пишется во множественном числе.

int lines[10];

vector<int> vertexes;

Имена переменных должны быть короткие и наглядные (имеющие ясную цель). Однобуквенные переменные не должны использоваться. В виде исключения – временные переменные. Общепринятые имена временных переменных являются:

i, j, k, m, n – для целых значений

c, d, e – для символов

Типы данных

Простые (баовые) типы данных и состоят из одного слова и пишутся маленькими буквами

int, vector, aabb, line

Классы

Названия классов пишутся с большой буквы. В многословных идентификаторах каждое слово пишется с большой буквы, остальные буквы - маленькие. Если одно из слов является аббревиатурой, то слово пишется так же как и в оригинальном варианте, и следующее слово отделяется от этого при помощи знака подчёркивания.

class MyBestClass

class SuperDVD_Writer

class _3D_Engine

Функции

Первое слово в имени функции (или метода) начинается с большой буквы и является глаголом (Generate, Is, Get, Set, Calculate). Каждое следующее слово пишется с большой буквы. Слова пишутся слитно.

Допускается написание первого слова с маленькой буквы, если первое слово маленькой длины (до трёх букв, например set get is was)

Параметры функций отделяются запятой, после которой желательно ставить пробел.

Пример:

void myBestFunction(int param1, float param2);

void GenerateText();

definition и declaration функции должны совпадать до символа (учитывая описательный комментарий).

Имена объектов являются неявными и не должны использоваться в именах методов.

line.getLength();   // NOT: line.getLineLength();

Если функция не принимает никаких параметров, то не надо указывать (void).

bool CheckError();

Перед входными параметрами надо писать IN, перед выходными OUT (эти пустые макросы определены в steel.h). Перед входными и выходными надо писать IN OUT.

Амперсант и звёздочка пишется сразу после типа. Не забывайте писать const перед параметрами.

void make(IN const std::string& a, OUT std::string& b)

Namespace

Имена, соответствующие названиям именованной области видимости (namespace), пишутся маленькими буквами.

model::analyzer, io::iomanager, common::math::geometry

 

Объявление и использование указателей

Символ указателя ставится перед названием переменной. Название типа отделяется от названия переменной  одним пробелом.

Пример:

int *a, *b;

int func(char* c);

 

const

 

Соглашения о выражениях

Типы

     Части класса должны быть отсортированы в следующем порядке: public, protected и private. В каждом блоке вначале должны идти поля, потом методы.

Преобразования типов должно осуществляться явно.

floatValue = static_cast<float>(intValue);

// NOT: floatValue = intValue;

Переменные

Переменные должны быть проинициализированы там, где они объявляются.

Использование глобальных переменных должно быть минимизировано.

Переменные в классе никогда не должны декларироваться как public.

Ссылки и указатели должны иметь свой ссылочный знак рядом с идентификатором.

 

float *x; // NOT: float* x;

int &y = …;   // NOT: int& y = …;

Неявный тест на 0 не должен нигде использоваться кроме как в логических выражениях. “0” не должен использоваться вместо “NULL”, ‘\0’ или 0.0

if (nLines != 0)  // NOT: if (nLines)

if (value != 0.0) // NOT: if (value)

Циклы

Переменные должны быть инициализированы непосредственно перед циклом.

 

isDone = false;       // NOT: bool isDone = false;

while(!isDone)       //      …

{

  …                   //      while(!isDone)

}                           {

                      //        …

                      //      }

Разное

Константы с плавающей точкой должны всегда писаться с точкой и по крайней мере одним знаком после нее.

 

double total = 0.0;    // NOT:  double total = 0;

double speed = 3.0e8;  // NOT:  double speed = 3e8;

 

double sum;

:

sum = (a + b) * 10.0;
