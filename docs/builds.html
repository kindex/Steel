Builds (official)

О документе

В этом документе описывается порядок и правила сборки проекта (build).

Имена билдов

Общий вид имени билда:

project[N][:subproject[M]]-release[-build][(revision)][-comment]

 

Где:

project – глобальное имя проекта (steel, wurps, gerda)

N – глобальная версия проекта

 

Subproject – нужно, если есть значительные идейные отличия от основного проекта, например steel: demo и steel: intro.

M – версия подпроекта. Версии отличаются идейным содержанием.

 

release – контрольная точка подпроекта. Отличаются функциональностью.

build – номер релиза для каждой версии подпроекта начинается с 1. Релизы друг от друга отличаются количеством багов, ресурсами. Код сильно не меняется.

comment – которкий комментарий к билду, например, реализованная фишка.

 

Дополнительно к build ставится один из следующих идентификаторов, уточняющий стабильность проекта:

 

t – тестовая версия. Для внутреннего использования. Либо нумеруются, либо идентифицируются строкой. Пример t3, t-lights. У тестовых версий независимая нумерация.

a – альфа версия. Сырая версия.

b – бета версия. Готовый билд по мнению разработчиков

rc – release candidate. Готовый билд по мнению тестеров.

r – финальный билд по мнению пользователей.

f – final build. Не поддерживается, ошибки в нём не исправляются.

 

Примерная история билдов:

t1, t2, … , t-a2fix, t-bunny-bug,

a1, a2, … ,

b4, b5, b9,

rc6, rc7, rc10,

r8, r11

 

revision – номер ревизии кода в репозитории из trunk. Если билд был сделан из другой ветки, то дополнительно указывается ветка.

 

Пример:

steel2:demo13-2-r3(273)-lights : « Вторая версия steel, demo 13ой версии, второй релиз, третий финальный билд этого релиза. Демонстрирует освещение»

В именах файлов “:” заменяется на “–“. Пример: steel2-demo13-2-r3(273).rar

Внутренние Имена билдов

В данный момент, для работы внутри команды, используется следующее:

     Steel (revision).

                Например Steel(266) или просто (266).

     Demo 14-ps-2, как правило, для сокращения, Steel или Demo опускается, и остается только [:subproject[M]]-t

                Например Demo 03-1; 04_illuminate; 09_internal_demo_1 и так далее.

                                       

Порядок создания билдов

Любой член команды может делать t build.

a и b buildы делается по согласию главного программиста.

rc и билды делаются только по согласию главного программиста и главного геймдизайнера.

r build делается по согласию заказчика, и является точной копией соответствующего rc билда.

Бинарник

Для OS Windows бинарник полностью перекомпилируется в DevC++ с включенной оптимизацией.

Для t, a, b версий включается отладочная информация (debug режим).

Для rc и r версий отладочная информация отключается. Возможно также упаковка бинарника, вроде exepak. Дополнительно в релиз добавляются все необходимые нестандартные библиотеки (dll).

Структура каталогов

Внутри главной директории должны быть:

линк на запускаемый файл (exe, bat, lnk, softlink).

README.txt

LICENSE.txt

VERSION.txt

CHANGELOG.txt

не должно быть никаких логов (steel.log, log/*)

bin/ содержит все запускаемые файлы и библиотеки

data/ - ресурсы игры

conf/ - настройки

 

Ресурсы

Архиватор

Все архивы содержат только одну директорию, которая совпадает с именем архива.

Допустимые архиваторы: rar, tar.bz2
